The HiroLeague website is a fully static marketing site built on the [ctrimm/astro-genai-startup-theme](https://github.com/ctrimm/astro-genai-startup-theme) (MIT license), customized for HiroLeague. For setup, build, and deployment instructions see [Website setup & deployment](/build/website).

## Design decisions

The theme was chosen because it ships with the full stack we need — Astro, React, Tailwind, shadcn/ui, Framer Motion, and dark mode — without requiring a CMS or external data sources. The site is fully static: all pages are pre-rendered to HTML at build time with no server-side logic.

**ReactBits** ([DavidHDev/react-bits](https://github.com/DavidHDev/react-bits), 36k+ stars) is used for animated UI effects. Components are installed via the jsrepo CLI as copy-pasted source files, meaning zero runtime dependency overhead. They work as Astro islands with `client:load` or `client:visible` directives.

The following sections were removed from the base theme because they were not requested:

- `Pricing.tsx`
- `Testimonials.tsx`
- `FAQ.tsx`
- The example markdown page (`markdown-page.md`)

## Site structure

```
hiroleague-website/
  src/
    components/
      ui/                    # shadcn/ui primitives (from theme)
      reactbits/             # ReactBits components (copied via CLI)
      Header.tsx
      Hero.tsx
      Features.tsx
      Waitlist.tsx
      Footer.tsx
      ThemeToggle.tsx
    layouts/
      main.astro             # Base layout
    pages/
      index.astro            # Landing page
      blog/
        index.astro          # Blog listing
        why-we-are-building-hiroleague.astro
      privacy.astro
      terms.astro
      404.astro
    styles/
      global.css             # Tailwind v4 tokens, dark-mode variant, animations
```

## Sections

### Hero

- Animated headline using ReactBits `BlurText`

- CTA button that scrolls to the waitlist section

`Features.tsx` — Four feature cards hinting at what HiroLeague does.

### Waitlist

`Waitlist.tsx` — Email capture section with an input + "Notify Me" button. The form markup is UI-only. The backend integration (Brevo embedded form) is tracked in `docs/internal_todo.md`.

### Footer

`Footer.tsx` — Copyright line and social icons (YouTube, Instagram, X, LinkedIn, TikTok) using Lucide icons.

## Editing content

### Hero, features, waitlist

Edit the corresponding React component in `src/components/`. The copy is inline — no CMS or separate data file.

### Blog

Add new articles as `.astro` files inside `src/pages/blog/` following the pattern of `why-we-are-building-hiroleague.astro`. Then add an entry to the `blogPosts` array at the top of `src/pages/blog/index.astro` so the article appears on the listing page.

### Typography

Two Google Fonts are loaded in `src/layouts/main.astro` via `<link>` tags:

| Font | Role | Tailwind token |
|---|---|---|
| [Plus Jakarta Sans](https://fonts.google.com/specimen/Plus+Jakarta+Sans) | Headings (`h1`–`h6`) and UI elements | `font-sans` |
| [Lora](https://fonts.google.com/specimen/Lora) | Body text | `font-serif` |

Both are registered as CSS custom properties in `src/styles/global.css` under `@theme inline`:

```css
--font-sans: "Plus Jakarta Sans", ui-sans-serif, system-ui, sans-serif;
--font-serif: "Lora", ui-serif, Georgia, serif;
```

The base layer applies `font-serif` to `body` and `font-sans` to all heading elements. To override on a specific element, use the Tailwind utilities `font-sans` or `font-serif` directly on the component.

### Colors and brand

All colors are CSS custom properties in `src/styles/global.css`. There are two layers:

**Semantic UI tokens** (`:root` / `.dark`) — control shadcn/ui component states (backgrounds, borders, primary button fill, etc.):

```css
--background, --foreground, --primary, --accent, --muted, --border, ...
```

**Brand tokens** (`:root` / `.dark`) — control the amber-to-pink visual identity. To rebrand, edit only these values:

| Token | Light (default) | Dark (default) | Usage |
|---|---|---|---|
| `--brand` | amber-500 | amber-400 | logo, badges, text accents |
| `--brand-mid` | orange-500 | orange-400 | gradient midpoint |
| `--brand-deep` | red-500 | red-400 | gradient endpoint |
| `--brand-rose` | rose-600 | rose-400 | feature card 3→4 stop |
| `--brand-pink` | pink-500 | pink-400 | feature card 4 endpoint |
| `--brand-hover` | auto via `color-mix` | auto | button hover states |
| `--brand-mid-hover` | auto via `color-mix` | auto | button hover states |
| `--brand-blob` | amber-300 | amber-900 | hero background blobs |
| `--brand-mid-blob` | orange-300 | orange-900 | hero background blobs |
| `--brand-deep-blob` | red-300 | red-900 | hero background blobs |

All tokens are mapped to Tailwind utilities in `@theme inline` (e.g. `--color-brand: var(--brand)`), so components use classes like `from-brand`, `text-brand`, `bg-brand/10`, `shadow-brand/25` — no hardcoded palette names anywhere. Dark mode variants are handled automatically by the `.dark` overrides; no `dark:` color prefixes are needed in component markup.

### Dark mode

Theme state is stored in `localStorage` under the key `theme`. `ThemeToggle.tsx` reads and writes it. An inline `<script is:inline>` in `src/layouts/main.astro` applies the `.dark` class synchronously on `<html>` before first paint, preventing the light-to-dark flash on page load.
